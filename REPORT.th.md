# Mood of the Major — รายงานโปรเจกต์

*Saig-Project · กระดานคอร์กบอร์ดแชร์อารมณ์แบบไม่ระบุตัวตนสำหรับนักศึกษา สจล.*

---

## 1. ภาพรวม / แนวคิด

**Mood of the Major** คือเว็บแอปที่ให้นักศึกษามหาวิทยาลัย สจล. (KMITL) แชร์ความรู้สึกของตัวเอง
แบบ *ไม่ระบุตัวตน* บนกระดาน *คอร์กบอร์ด* ดิจิทัลร่วมกัน นักศึกษาแต่ละคนจะ "ปัก" โน้ตกระดาษโพสต์อิท
ใบเล็ก ๆ ที่ติดแท็กอารมณ์ (มีความสุข สงบ เหนื่อย เครียด เศร้า ตื่นเต้น โกรธ) พร้อมข้อความสั้น ๆ
โน้ตแต่ละใบจะแสดงด้วยสีตามอารมณ์ มีรูปหมุดปัก และเอียงแบบสุ่มเล็กน้อย ทำให้ทั้งกระดานดูเหมือน
บอร์ดคอร์กจริง ๆ

เป้าหมายคือการได้เช็กอินความรู้สึกโดยไม่มีแรงกดดันทางสังคม ชื่อจริงและรหัสนักศึกษาของคุณ
**จะไม่ถูกแสดง** เด็ดขาด แต่ทุกโพสต์จะลงชื่อด้วยนามแฝงที่ระบบสร้างให้แบบน่ารัก ๆ
(เช่น *"Sleepy Capybara #42"*) พร้อมคณะ สาขา และชั้นปี ซึ่งเพียงพอที่จะเห็นแนวโน้มอารมณ์
ทั่วทั้งมหาวิทยาลัย แต่ไม่มากพอที่จะระบุตัวใครได้

ตัวผลิตภัณฑ์มีหน้าจอหลัก 3 ส่วน:

- **กระดาน (The Board)** — ฟีดสาธารณะของโน้ตอารมณ์แบบไม่ระบุตัวตน พร้อมตัวกรองและการแบ่งหน้า
- **Campus Vibe (สถิติ)** — การแสดงผลแบบสดของการกระจายตัวของอารมณ์ทั่วมหาวิทยาลัย รวมถึง
  "อารมณ์ที่พบมากที่สุด" ในตอนนี้ โดยพื้นหลังของหน้าจะเปลี่ยนโทนสีตามอารมณ์นั้น
- **การกลั่นกรอง (Admin)** — แผงควบคุมส่วนตัวที่ผู้ดูแลระบบสามารถตรวจสอบและลบโน้ตใดก็ได้

การเข้าถึงถูกควบคุมด้วยขั้นตอน **การยืนยันความเป็นนักศึกษา สจล.** แบบเบา ๆ ตอนสมัครสมาชิก
เพื่อให้กระดานนี้เป็นพื้นที่สำหรับนักศึกษาเท่านั้น

## 3. การจับคู่ฟีเจอร์กับข้อกำหนดทีละข้อ

คำอธิบายสัญลักษณ์: ✅ ทำเสร็จ · ➕ โบนัสที่ทำเสร็จ · ⛔ โบนัสที่ยังไม่ได้ทำ (บันทึกไว้เป็นงานต่อเนื่อง)

### ✅ การยืนยันตัวตนและระบุตัวตน (สมัคร / เข้าสู่ระบบ + การยืนยัน สจล.)
- การสมัครจะเก็บรหัสนักศึกษา อีเมล สจล. คณะ สาขา ชั้นปี และรหัสผ่าน
  (`POST /api/auth/register`) รหัสผ่านถูกเข้ารหัสด้วย **bcrypt (12 rounds)** — ไม่มีการเก็บ
  รหัสผ่านแบบข้อความล้วนเด็ดขาด
- **การยืนยันความเป็นนักศึกษา สจล.** ทำงานตอนสมัครผ่าน `server/src/services/kmitlVerify.js`:
  - หากตั้งค่า `KMITL_API_KEY` ไว้ ระบบจะเรียก KMITL Developer Platform API
    (`/students/profile`) เพื่อยืนยันว่านักศึกษามีตัวตนจริงและรายละเอียดตรงกัน
  - หาก **ไม่มีคีย์** (ค่าเริ่มต้น ดูหัวข้อ §7) ระบบจะใช้ **การตรวจสอบรูปแบบอย่างเข้มงวด**:
    รหัสนักศึกษาต้องเป็นตัวเลข 8 หลักพอดี อีเมลต้องเป็น `<รหัสนักศึกษา>@kmitl.ac.th`
    และตัวเลขสองหลักแรกของรหัส (คำนำหน้าปีที่เข้าเรียนตามปฏิทินพุทธศักราช) ต้องสอดคล้อง
    กับชั้นปีที่แจ้ง (ยืดหยุ่น ±1 ปีสำหรับรอยต่อของปีการศึกษา)
- **เซสชัน JWT พร้อมการหมุนเวียน refresh token:** access token อายุสั้น (15 นาที) ส่งกลับให้
  ฝั่งไคลเอนต์ และ refresh token เก็บเป็นคุกกี้แบบ **httpOnly** โดย
  `POST /api/auth/refresh` จะหมุนเวียน refresh token และออก access token ใหม่ ระบบเก็บ
  ค่าแฮชของ refresh token *ปัจจุบัน* ไว้กับผู้ใช้ ดังนั้นหากมีการนำ token ที่ถูกขโมย/ใช้ซ้ำ
  มาใช้ จะทำให้เซสชันถูกเพิกถอนทั้งหมด ส่วน `POST /api/auth/logout` จะล้างคุกกี้และเพิกถอน token
- `GET /api/auth/me` คืนโปรไฟล์ของผู้ใช้ปัจจุบัน

### ✅ CRUD ของกระดานอารมณ์
- **สร้าง** — `POST /api/moods` (ต้องยืนยันตัวตน): ประเภทอารมณ์ + ข้อความ
- **อ่าน** — `GET /api/moods` (ฟีดสาธารณะ) และ `GET /api/moods/:id`
- **แก้ไข** — `PATCH /api/moods/:id` (**เจ้าของเท่านั้น**)
- **ลบ** — `DELETE /api/moods/:id` (**เจ้าของหรือผู้ดูแลระบบ**)

### ✅ การแชร์แบบไม่ระบุตัวตน
- ผู้ใช้ทุกคนจะได้รับนามแฝงแบบสุ่มที่ย้อนกลับไม่ได้ตอนสมัคร
  (`aliasService.js` เช่น *"Cheerful Otter #128"*)
- โมเดล `Mood` เก็บ id เจ้าของจริง **ไว้ภายในเท่านั้น** (สำหรับตรวจสอบความเป็นเจ้าของ) และ
  บันทึกสแนปช็อตแบบไม่ระบุตัวตน (นามแฝง คณะ สาขา ชั้นปี) ณ ตอนที่โพสต์ ตัวจัดรูปแบบ
  `toPublicJSON()` รับประกันว่าชื่อจริง/รหัสนักศึกษา **จะไม่ปรากฏ** ในการตอบกลับ API ใด ๆ

### ✅ การค้นหา — ค้นหา / กรอง / แบ่งหน้า
- `GET /api/moods` รับพารามิเตอร์ `moodType`, `faculty`, `major`, `dateFrom`, `dateTo`,
  `page` และ `limit` การแบ่งหน้าฝั่งเซิร์ฟเวอร์ใช้ `skip`/`limit` พร้อมยอดรวมจาก
  `countDocuments` และการตอบกลับจะมีบล็อก `pagination` (`page`, `limit`, `total`, `totalPages`)
- ฟิลด์ที่ทำดัชนี (`moodType`, `authorFaculty`, `authorMajor`, `createdAt`) ช่วยให้การกรองเร็ว

### ✅ RBAC (การควบคุมการเข้าถึงตามบทบาท)
- มีสองบทบาท: `student` และ `admin`
- มิดเดิลแวร์ `authenticate` ตรวจสอบ JWT ส่วน `authorize('admin')` ป้องกันเส้นทางของแอดมิน
- ความเป็นเจ้าของถูกตรวจสอบฝั่งเซิร์ฟเวอร์เสมอจากฟิลด์ `author` ที่ซ่อนอยู่ ผู้ที่ไม่ใช่เจ้าของ
  จะได้รับ **403** เมื่อพยายามแก้ไข มีเพียงเจ้าของหรือแอดมินที่ลบได้ และรายการกลั่นกรองของแอดมิน
  (`GET /api/admin/moods`) จะคืน **403** สำหรับผู้ที่ไม่ใช่แอดมิน

### ✅ การตรวจสอบฟอร์ม (ทั้งไคลเอนต์และเซิร์ฟเวอร์ สองชั้น)
- **เซิร์ฟเวอร์:** `express-validator` บนทุกเส้นทาง — enum อารมณ์ที่ถูกต้อง ความยาวข้อความ
  1–280 การตรวจสอบ Mongo ID รหัสนักศึกษา 8 หลัก กฎอีเมล `<รหัสนักศึกษา>@kmitl.ac.th`
  ชั้นปี 1–8 รหัสผ่าน ≥ 8 ตัวอักษร และขอบเขตการแบ่งหน้า
- **ไคลเอนต์:** schema ของ **Zod** ที่สอดคล้องกัน ผ่าน React Hook Form
  (`client/src/lib/schemas.ts`) รวมถึงการยืนยันรหัสผ่านและกฎอีเมล/รหัสแบบเดียวกัน ทำให้ผู้ใช้
  ได้รับข้อเสนอแนะทันทีก่อนที่จะส่งคำขอด้วยซ้ำ

### ➕ เอกสาร Swagger / OpenAPI (โบนัส)
- **Swagger UI แบบโต้ตอบได้ที่ `/api-docs`** (และสเปกดิบที่ `/api-docs.json`) สร้างด้วย
  `swagger-jsdoc` จากคำอธิบาย JSDoc บนทุกเส้นทาง รวมถึง security scheme แบบ Bearer-auth
  เพื่อให้ทดลองเรียกเอนด์พอยต์ที่ต้องยืนยันตัวตนได้ในเบราว์เซอร์

### ✅ การจัดการสถานะ (state management)
- React Context: `AuthContext` (เซสชัน + login/register/logout พร้อมกู้คืนเซสชันด้วยการ
  refresh แบบเงียบตอนโหลด) และ `MoodContext` (ข้อมูลกระดาน ตัวกรอง การแบ่งหน้า และ
  create/edit/delete แบบ optimistic) ตัว axios client เก็บ access token **ไว้ในหน่วยความจำ
  เท่านั้น** (ปลอดภัยจาก XSS มากขึ้น) และลองส่งคำขอซ้ำอัตโนมัติหนึ่งครั้งหลัง refresh เมื่อเจอ 401

### ➕ แอนิเมชัน (โบนัส)
- ใช้ **Framer Motion** ทั่วทั้งแอป: โน้ตโพสต์อิทเด้งเข้ามาพร้อมหมุนเล็กน้อย มีแอนิเมชันตอน hover
  และใช้ layout/`AnimatePresence` เมื่อเปลี่ยนตัวกรอง แถบสถิติมีแอนิเมชันความกว้าง และ
  โมดัลเขียนโน้ตมีแอนิเมชันตอนเปิด

### ✅ สถาปัตยกรรม / คุณภาพโค้ด
- ฝั่งหลังบ้านแบ่งเป็นชั้นอย่างเป็นระเบียบ: `config`, `models`, `controllers`, `services`,
  `routes`, `middleware`, `validators`, `docs`, `utils`, `seed` มีการจัดการข้อผิดพลาดรวมศูนย์
  (`ApiError` + `errorHandler`) ตัวห่อ `asyncHandler` ส่วนหัวความปลอดภัยจาก Helmet และ
  รายการอนุญาต CORS อย่างเข้มงวด (พร้อมข้อยกเว้นสำหรับ localhost ตอนพัฒนา)

### ➕ ความคิดสร้างสรรค์ / การแสดงผลข้อมูล (โบนัส)
- ธีมคอร์กบอร์ด (พื้นผิว หมุด ฟอนต์ลายมือ สีตามอารมณ์ การเอียงแบบคงที่) หน้า **สถิติ "Campus
  Vibe"** พร้อมตัวบ่งชี้อารมณ์ที่พบมากที่สุด แถบอารมณ์แยกตามคณะ และพื้นหลังหน้าที่ไล่โทนสี
  ตามอารมณ์หลักของมหาวิทยาลัย

### ⛔ การดีพลอยขึ้นคลาวด์ (โบนัส — ยังไม่ได้ทำ)
- ยังไม่มีการดีพลอยขึ้น Vercel/Render เก็บไว้เป็นงานต่อเนื่องอย่างชัดเจน (ดูหัวข้อ §7)
  ตัวแอปรันได้ครบถ้วนในเครื่อง

## 4. สรุป API

Base URL (ในเครื่อง): `http://localhost:4000` · ทุกเส้นทางอยู่ภายใต้ `/api`
เอกสารฉบับเต็มแบบโต้ตอบได้ที่ **`http://localhost:4000/api-docs`**

| Method | เอนด์พอยต์ | สิทธิ์ | วัตถุประสงค์ |
|---|---|---|---|
| GET | `/api/health` | — | ตรวจสอบสถานะระบบ |
| POST | `/api/auth/register` | — | สร้างบัญชี (พร้อมยืนยัน สจล.) |
| POST | `/api/auth/login` | — | เข้าสู่ระบบ เริ่มเซสชัน |
| POST | `/api/auth/refresh` | คุกกี้ | หมุนเวียน refresh token ออก access token ใหม่ |
| POST | `/api/auth/logout` | คุกกี้ | เพิกถอนเซสชัน |
| GET | `/api/auth/me` | Bearer | โปรไฟล์ผู้ใช้ปัจจุบัน |
| POST | `/api/moods` | Bearer | ปักโน้ตอารมณ์ใหม่ |
| GET | `/api/moods` | ไม่บังคับ | ฟีดพร้อมตัวกรอง + แบ่งหน้า |
| GET | `/api/moods/:id` | ไม่บังคับ | โน้ตใบเดียว |
| PATCH | `/api/moods/:id` | Bearer (เจ้าของ) | แก้ไขโน้ตของตัวเอง |
| DELETE | `/api/moods/:id` | Bearer (เจ้าของ/แอดมิน) | ลบโน้ต |
| GET | `/api/stats` | Bearer | การกระจายอารมณ์ + อารมณ์หลักของมหาวิทยาลัย |
| GET | `/api/admin/moods` | Bearer (แอดมิน) | รายการกลั่นกรอง (403 สำหรับผู้ที่ไม่ใช่แอดมิน) |

การตอบกลับทั้งหมดใช้รูปแบบเดียวกัน: `{ success, data, message? }`

สิ่งที่ต้องมี: **Node.js 20+** และ **Docker Desktop** (สำหรับ MongoDB) หรือ connection string
ของ MongoDB อื่นใดก็ได้ *(ณ ขณะที่เขียนรายงานนี้ ได้ยืนยันแล้วว่าทั้ง API ที่พอร์ต 4000 และ
ไคลเอนต์ Vite ที่พอร์ต 5173 กำลังทำงานอยู่โดยเชื่อมต่อกับ MongoDB ในเครื่องแบบสด)*

```bash
# 1. ติดตั้ง dependencies ของทั้งสองเวิร์กสเปซ
npm run install:all           # = npm install ใน server/ และ client/

# 2. เริ่ม MongoDB (Docker)
npm run docker:up             # mongo:7 ที่ 127.0.0.1:27017

# 3. ตั้งค่า environment ของเซิร์ฟเวอร์
#    คัดลอก server/.env.example -> server/.env  (ค่าเริ่มต้นใช้ได้กับการพัฒนาในเครื่อง)

# 4. สร้างบัญชีแอดมินเริ่มต้น (อ่านค่า ADMIN_* จาก server/.env)
npm run seed:admin --prefix server

# 5. รันฝั่งหลังบ้าน  (http://localhost:4000, Swagger ที่ /api-docs)
npm run dev:server

# 6. รันฝั่งหน้าบ้าน (http://localhost:5173)
npm run dev:client
```

พอร์ตเริ่มต้น: **API `4000`**, **ไคลเอนต์ `5173`**, **MongoDB `27017`** ฝั่งไคลเอนต์อ่าน URL
ฐานของ API จาก `VITE_API_URL` (ค่าเริ่มต้นคือ `http://localhost:4000`) ส่วนเซิร์ฟเวอร์อ่าน
การตั้งค่าจาก `server/.env` (ดู `server/.env.example`)

## 6. ประวัติการทดสอบ / การตรวจสอบ

มีการทดสอบแบบครบวงจร (end-to-end) กับระบบที่กำลังรันอยู่ — ทั้งผ่าน API และการเดินผ่าน
ทีละหน้าในเบราว์เซอร์ด้วยตนเอง:

- **การยืนยันตัวตน:** สมัคร (พร้อมยืนยันรูปแบบ สจล.) เข้าสู่ระบบ refresh เซสชัน และออกจากระบบ
- **CRUD ของอารมณ์:** สร้าง อ่าน แก้ไข ลบ — โดยมีการบังคับ **ตรวจสอบความเป็นเจ้าของ**
- **RBAC:** ยืนยันการตอบกลับ **403** เมื่อผู้ที่ไม่ใช่เจ้าของพยายามแก้ไข/ลบโน้ตของผู้อื่น และ
  เมื่อผู้ที่ไม่ใช่แอดมินเรียกเอนด์พอยต์กลั่นกรองของแอดมิน
- **การค้นหา:** การกรองตาม `moodType`, `faculty` และช่วงวันที่ พร้อมการแบ่งหน้าข้ามหลายหน้า
- **สถิติ:** `GET /api/stats` คืนการกระจายตัวและอารมณ์หลัก หน้า Campus Vibe แสดงผลและ
  ไล่โทนสีได้อย่างถูกต้อง
- **เอกสาร:** Swagger UI ที่ `/api-docs` แสดงทุกเอนด์พอยต์และสามารถทดลองเรียกใช้ได้
- **UI:** เดินผ่านทุกหน้าในเบราว์เซอร์ ได้แก่ กระดาน โมดัลเขียนโน้ต สถิติ และหน้าแอดมิน

**บั๊กที่พบและแก้ไขแล้ว:** **ตารางกลั่นกรองของแอดมินตัดปุ่ม Delete หายไป** เมื่อความกว้าง
หน้าจอต่ำกว่าประมาณ 950px เพราะคอนเทนเนอร์ของตารางใช้ `overflow-hidden` ได้แก้เป็น
`overflow-x-auto` พร้อมกำหนด `min-w` ให้ตาราง เพื่อให้แถวเลื่อนแนวนอนแทนที่จะซ่อนปุ่ม
และได้ยืนยันการแก้ไขแบบสดอีกครั้ง *(สถานะปัจจุบันใน `client/src/pages/AdminPage.tsx`:
ตัวห่อใช้ `overflow-x-auto overflow-y-hidden` และตารางมี `min-w-[640px]`)*

## 7. ข้อจำกัดที่ทราบและขั้นตอนถัดไปที่แนะนำ

1. **การยืนยันแบบสด / SSO ของ สจล.** **ยังไม่ได้** รับคีย์ API สดของ สจล. — พอร์ทัลนักพัฒนา
   ของ สจล. (`developer.kmitl.ac.th`) ต้องใช้ข้อมูลรับรองที่ผ่านการอนุมัติจากผู้ดูแล ปัจจุบัน
   แอปใช้ **การตรวจสอบรูปแบบอย่างเข้มงวด** ตามที่อธิบายในหัวข้อ §3 โค้ดถูกเขียนให้สลับได้ง่าย:
   เพียงตั้งค่า `KMITL_API_KEY` (และหากต้องการ `KMITL_API_BASE_URL`) ใน `server/.env`
   บริการยืนยันจะสลับไปใช้ Lookup API สดโดยอัตโนมัติ — ไม่ต้องแก้โค้ดส่วนอื่น ขั้นตอนถัดไป
   คือการทำ KMITL SSO/OAuth เต็มรูปแบบเมื่อได้รับข้อมูลรับรอง

2. **การเผยแพร่บน GitHub** ยังไม่มีการสร้าง repository สาธารณะบน GitHub ติดอยู่ที่ขั้นตอน
   `gh auth login` แบบโต้ตอบ ซึ่ง **เจ้าของโปรเจกต์ต้องรันด้วยตนเอง** เมื่อยืนยันตัวตนแล้ว
   จึงจะสร้าง repo และ push ได้

3. **การดีพลอยขึ้นคลาวด์** ยังไม่ได้ดีพลอยขึ้น Vercel/Render งานต่อเนื่องที่แนะนำ: ดีพลอย API
   พร้อม MongoDB แบบ managed (เช่น MongoDB Atlas) ดีพลอยไคลเอนต์เป็น static build และตั้ง
   ค่า env สำหรับ production (`CLIENT_URL`, JWT secrets, `MONGODB_URI`, `KMITL_API_KEY`)

4. **สถานะการควบคุมเวอร์ชัน** ใน git มีเพียง **คอมมิตโครงสร้างเริ่มต้น (bootstrap-skeleton)**
   เท่านั้น ส่วนโค้ดที่พัฒนาเต็มรูปแบบยัง **ไม่ได้คอมมิตและอยู่ใน working tree** การคอมมิต
   (แล้วจึงเผยแพร่ตามข้อ 2) เป็นขั้นตอนถัดไปโดยธรรมชาติ

## 8. ข้อมูลบัญชีแอดมิน (บัญชีทดสอบสำหรับการพัฒนาเท่านั้น)

สำหรับการตรวจให้คะแนน/ทดสอบ มีบัญชีแอดมินเริ่มต้นที่สร้างโดย `npm run seed:admin`:

| ฟิลด์ | ค่า |
|---|---|
| รหัสนักศึกษา | `99999999` |
| อีเมล | `99999999@kmitl.ac.th` |
| รหัสผ่าน | `Admin123!` |

> **หมายเหตุ:** นี่คือ **ข้อมูลรับรองสำหรับการพัฒนาเท่านั้น** ที่กำหนดไว้ใน `server/.env.example`
> ต้องเปลี่ยน (หรือลบบัญชีนี้) ก่อนการดีพลอยจริงใด ๆ
